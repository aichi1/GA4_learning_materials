# セッション・ユーザー・エンゲージメントの概念

## この章の目標

- GA4 における「セッション」「ユーザー」の定義を正確に理解する
- エンゲージメントセッション vs バウンス（直帰）の違いを把握する
- DebugView の見方を理解し、handson の準備を整える

---

## 1. ユーザーとは

GA4 でいう「ユーザー」とは、サイトを訪問した人（ブラウザ）を指します。

### 識別方法

| 識別子 | 説明 |
|--------|------|
| クライアントID | ブラウザの Cookie に保存される識別子。同じブラウザからの訪問を「同一ユーザー」とみなす |
| ユーザーID | ログインシステムと連携して付与できる識別子（Week 2 以降） |

### 新規ユーザー vs リピーター

- **新規ユーザー**: そのブラウザで初めてサイトを訪問した人（`first_visit` イベントが発火）
- **リピーター**: 過去に訪問歴がある人

Cookie をクリアしたり、別ブラウザを使うと「新規ユーザー」としてカウントされます。
これはGA4の計測上の限界であり、完全な個人識別ではありません。

---

## 2. セッションとは

セッションとは、ユーザーの「ひとまとまりの訪問」を指します。

### GA4 のセッション定義

GA4 では、以下のいずれかの条件でセッションが開始されます。

- ユーザーが最初にサイトに訪問したとき
- 前回のセッション開始から30分以上経過した後に新たにページを開いたとき

UAと違い、日付をまたいでもセッションは継続します（30分ルールのみ）。

### セッションID

各セッションには一意のIDが付与され、そのセッション内のすべてのイベントに紐づきます。

---

## 3. エンゲージメントセッションとは

GA4 では単なる「セッション数」ではなく、**質の高い訪問（エンゲージメントセッション）** を重視します。

### エンゲージメントセッションの条件（以下のいずれかを満たすこと）

| 条件 | 説明 |
|------|------|
| 滞在時間 10秒以上 | ページを10秒以上閲覧した |
| コンバージョンイベント発生 | 購入・フォーム送信などが発生した |
| 2ページ以上閲覧 | 複数ページを遷移した |

### エンゲージメント率

```
エンゲージメント率 = エンゲージメントセッション数 ÷ 総セッション数 × 100
```

エンゲージメント率が高いほど、サイトが訪問者の興味を引けていることを示します。

---

## 4. バウンスとは（GA4 での再定義）

UAでは「1ページだけ見て離脱 = バウンス」でしたが、GA4 では定義が変わりました。

| 指標 | GA4 での定義 |
|------|-------------|
| バウンス | エンゲージメントセッションにならなかった訪問（10秒未満・1ページのみ・CVなし） |
| バウンス率 | 全セッションのうちバウンスの割合 |

**考え方の変化**:
- UA: 「1ページしか見なかった = 失敗」
- GA4: 「10秒以上見ていれば価値ある訪問とみなす」

ブログ記事のように1ページで完結するコンテンツは、GA4のバウンス率ではなくエンゲージメント率で評価するのが適切です。

---

## 5. ページビューとスクリーンビュー

| イベント名 | 発火タイミング |
|-----------|--------------|
| `page_view` | ページが読み込まれたとき（URLが変わったとき） |
| `screen_view` | モバイルアプリで画面が切り替わったとき |

### SPA（Next.js）での注意点

通常のウェブサイトはページ遷移のたびにブラウザが再読み込みされるため、
`page_view` が自動的に発火します。

しかし Next.js のような SPA（Single Page Application）では、
**URLが変わってもブラウザは再読み込みされません**。
そのため、URLの変化を JavaScript で監視して、手動で `page_view` イベントを送信する必要があります。

この実装を handson で行います。App Router では `usePathname` フックを使います。

---

## 6. DebugView とは

DebugView は、GA4 管理画面内にあるリアルタイム確認ツールです。

### できること

- 送信されたイベントをリアルタイムで確認できる
- イベントのパラメータ（どんなデータが送られたか）を詳細に見られる
- 本番データに影響を与えずにテストできる（デバッグモードが必要）

### 通常のリアルタイムレポートとの違い

| 機能 | リアルタイムレポート | DebugView |
|------|---------------------|----------|
| 対象 | 全ユーザー | デバッグモード有効デバイスのみ |
| イベント詳細 | 限定的 | すべてのパラメータを確認可能 |
| 用途 | 全体の動向確認 | 実装の動作確認・デバッグ |

### DebugView の有効化方法

ブラウザの URL に `?_ga4_debug=1` を追加するか、
Chrome 拡張「Google Analytics Debugger」を使うと、
そのデバイスのセッションだけが DebugView に表示されます。

## 7. GA4 の主要指標まとめ

| 指標 | 説明 |
|------|------|
| ユーザー数 | 計測期間中の訪問ユーザー数 |
| 新規ユーザー数 | 初めてサイトを訪問したユーザー数 |
| セッション数 | 訪問の回数（1ユーザーが複数セッション持てる） |
| エンゲージメントセッション数 | 質の高い訪問の回数 |
| エンゲージメント率 | セッション中でエンゲージメントセッションの割合 |
| エンゲージメント時間 | ページがフォアグラウンドにあった合計時間 |
| イベント数 | 全イベントの発火回数 |
| コンバージョン数 | コンバージョンとして設定したイベントの発火回数 |

---

## まとめ

- ユーザーはブラウザCookieで識別される。完全な個人識別ではない
- セッションは「30分の非活動」でリセット。日付をまたいでも継続する
- エンゲージメントセッション = 10秒以上 or 2ページ以上 or CV発生。これがGA4の質の評価軸
- バウンスはGA4で再定義された。エンゲージメントなし訪問 = バウンス
- SPA（Next.js）では `usePathname` でURL変化を監視して `page_view` を手動送信する必要がある
- DebugView でリアルタイムに自分のイベントだけを確認できる

次は practice で理解度を確認し、その後 handson で実際に実装します。
